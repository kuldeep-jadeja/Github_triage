## P1 — Must Do Before Production

### Migrate from SQLite to PostgreSQL
- **What:** Replace SQLite with PostgreSQL for concurrent write support and better query performance.
- **Why:** SQLite serializes writes — becomes a bottleneck at 50+ concurrent triages. PostgreSQL supports concurrent writes, better indexing, and replication.
- **Pros:** Scales to production load, supports multiple instances, better query performance for metrics.
- **Cons:** Adds operational complexity (DB hosting, migrations, connection pooling).
- **Context:** Current plan uses SQLite for hackathon simplicity. The schema is defined in the review (Section 9). Migration path: dump SQLite → import to PostgreSQL, update connection string.
- **Effort estimate:** M (human team) → with CC+gstack: S
- **Priority:** P1
- **Depends on:** None

### GitHub App Migration
- **What:** Upgrade from Personal Access Token to full GitHub App authentication (manifest flow, private key management, installation tokens).
- **Why:** PATs have broader permissions than needed and can't be scoped to specific repos. GitHub Apps are the production-ready approach.
- **Pros:** Fine-grained permissions, per-repo installation, better security, marketplace-ready.
- **Cons:** Adds setup complexity (manifest flow, private key rotation), more code paths to test.
- **Context:** PAT accepted for hackathon simplicity. The plan already has GitHub App code — just needs to be the default instead of PAT.
- **Effort estimate:** M (human team) → with CC+gstack: S
- **Priority:** P2
- **Depends on:** None (can be done in parallel)
